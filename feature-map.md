# LLMGateway 产品地图

本文是创业团队共同使用的产品总纲：说明 LLMGateway 服务谁、解决什么问题、靠什么能力交付、如何形成收入、哪些风险必须守住，以及未来按什么原则演进。实现细节以 `spec.md` 和 `architecture.md` 为准，当前完成度以 `plan.md` 为准。

## 一句话定位

LLMGateway 是服务约 200～300 名受控用户的中小微企业级多 Provider LLM 网关。公司统一采购、接入和运营多个上游模型，客户或内部成员只拿一个 Gateway Key，就能稳定调用被授权的模型。

产品围绕统一中转、用户管理和最低生产地基持续演进，安全、额度、并发、错误、恢复和运维达到真实生产标准。

## 服务对象

- 管理员：创业团队中负责 Provider、凭据、模型目录、用户、额度和运行健康的人。
- 成员：公司员工、合作团队或签约客户中获准使用模型 API 的人。
- 调用方：成员使用的 SDK、Agent、IDE 插件、自动化任务或内部业务系统。
- 所有使用者都经过邀请、审核和授权；成员只接触 Gateway Key。

## 客户价值

- 一个 Base URL 和一个 Gateway Key 调用多个 Provider，不让每个团队重复接入厂商差异。
- 上游凭据集中保管，人员流动时只撤销网关权限，不散落真实 Provider Key。
- 管理员能控制谁可以用什么模型、能用多少、能并发多少，并看见真实 usage。
- 单个 Provider 或凭据限流、冷却或失败时，网关在安全边界内选择其他合格资源。
- 排队、错误、取消、断连、强杀和重启都有明确结果，不靠人工猜测是否扣费或是否已发送。

## 核心一：多 Provider 统一中转

### Provider 与目录

- 管理员创建 Provider、模型和加密凭据，声明模型是否支持流式、工具、reasoning 和结构化输出。
- reasoning 模型还要声明 `toggle`、`effort` 或 `hybrid` 控制 profile；同一个 OpenAI-compatible adapter 据此表达模型差异，普通请求不会因上游默认 thinking 吞掉全部可见输出预算。
- 凭据绑定允许的模型、免费/付费资源域、priority、weight、RPM、TPM 和并发。
- 编辑中的 live registry 不直接影响用户；管理员捕获不可变 revision，校验后发布，数据面只读取当前 active revision。
- 当前专用 adapter 为智谱 GLM、Agnes 和 Google Gemini，同时支持明确能力子集的通用 OpenAI-compatible Provider。

### 统一 API

- `GET /v1/models` 只返回当前 Key 有权使用且已发布的模型。
- `POST /v1/chat/completions` 和 `POST /v1/responses` 支持非流、流式、工具、reasoning 和 usage。
- Provider 差异在 adapter 内转换；无法可靠转换的能力在发送前拒绝，不静默丢字段或伪造支持。

### Provider 扩展合同

Provider 集合必须持续更新，但扩展不能让协议、调度和账本跟着厂商分叉。新增接入按以下顺序完成：

1. 先核验官方 API、鉴权、模型名、能力、流事件、usage、错误、限额、服务条款和 SDK/wire；记录版本与证据。
2. 能由 OpenAI-compatible 明确子集无损表达时，只新增 Provider 配置和模型能力，不创建专用 adapter。
3. 确有无法无损表达的差异时，实现 `providers.Adapter` 的请求、非流响应、流解析、错误分类、探测和能力合同。
4. 在唯一 Provider catalog 注册 kind、展示名称和 builder；请求执行、凭据探测、写入校验和管理端类型列表会消费同一事实。
5. 用隔离 Provider 覆盖非流、流、工具、reasoning、usage、429/5xx、畸形响应、取消与发送边界，再进入真实管理员发布和成员调用旅程。
6. 同步 `spec.md`、产品地图、adapter 文档和验证证据；未经核验的能力保持不支持。

Provider catalog 编译进受审查的版本发布，持续更新的同时守住 secret、网络和供应链边界。

### 路由策略

1. 先做硬资格过滤：目录已发布、模型匹配、资源域一致、能力满足、凭据启用且已授权、没有冷却、不是本次已经失败的 attempt。
2. 在合格凭据中选择数值最小的管理员 priority。
3. 同 priority 内按正 weight 分配流量，让管理员表达主备或比例，而不是依赖不可解释的综合分数。
4. RPM、TPM 和并发由 Valkey 原子计数与租约控制；额度由 PostgreSQL 原子账本控制，不参与虚构“健康分”。
5. 免费池和付费池严格隔离，免费资源不可用时不自动消耗付费资源。

### 公平、失败与恢复

- 单实例内每个用户自己的请求保持 FIFO，用户之间轮转，并限制单用户占用的活动槽位。
- 多实例共享全局和各维度容量，但不承诺进程重启后恢复绝对排队位置。
- 429、临时 5xx 和超时会更新凭据冷却；只有确认未产生未知副作用时才有限重试。
- 流已经输出后不拼接第二个 Provider；发送结果未知时记录 `uncertain`，保留额度预留并等待恢复判断。
- 客户端取消、断连、后台 Responses、进程强杀和重启都有持久状态与幂等恢复 owner。
- 当前 300 用户 profile 以约 60 人持续活跃为容量基线；稳态流量不得依赖初始令牌桶掩盖不可持续 RPM，300 人同步突发允许按硬并发边界返回带恢复信息的 429。

## 核心二：用户与管理员

### 加入与身份

- 系统只有管理员和成员两种角色。
- 第一位管理员通过一次性初始化建立；成员使用管理员签发的邀请码注册，审核通过后才能调用。
- 管理会话有过期、注销、HttpOnly cookie 和 CSRF 防护；停用用户会阻止后续管理与 API 调用。

### 授权与 Key

- 管理员为成员分配模型范围、额度、RPM、TPM、并发和到期时间，并签发或撤销 Gateway Key。
- 成员只查看自己的授权、余额、usage 和 Key，只能更换或撤销自己的 Key。
- 完整 Gateway Key 和邀请码只在创建或同一幂等操作恢复时展示；数据库只保存摘要和展示前缀。
- 管理员可幂等重置成员密码并撤销旧会话，也可保留当前会话撤销自己的其他登录；唯一管理员锁定由带确认开关、password file 和 system audit 的离线命令恢复。
- Key 更换先创建同 owner、授权与到期时间的新 Key，新旧 Key 同时可用；客户端切换确认后再撤销旧 Key。

### 额度与用量

- 请求在发送前原子预留额度，完成后使用上游权威 usage 结算；没有权威值时明确使用估算来源。
- 并发请求不能超扣，重复回调或恢复不能重复结算；失败释放、补偿和不确定预留都有审计来源。
- 管理员能查看全局用量和成员用量，成员只能查看自己的记录。

## 最低生产地基

| 边界 | 必须得到的结果 |
| --- | --- |
| Secret | Provider 凭据使用版本化 AEAD；Key、邀请、会话和 CSRF 只存摘要 |
| 权限 | 服务端强制管理员/成员最小权限，UI 隐藏不能代替授权 |
| 网络 | 自定义 Provider URL 防御 SSRF、DNS 重绑定、恶意重定向和内网访问 |
| 资源 | 请求体、响应字节、队列、连接、流时长、RPM/TPM 和并发都有上限 |
| 持久 | PostgreSQL 原子拥有用户、配置、请求、attempt、额度、usage 和审计 |
| 协调 | Valkey 只保存可过期租约与限流计数；不可用时 fail closed |
| 隐私 | 日志、错误、指标、审计和浏览器持久状态不保存 secret 或请求正文 |
| 恢复 | 取消、断连、partial stream、强杀、重启和多实例有明确终态与 owner |
| 运维 | Linux 双 Gateway/Caddy TLS Compose 与 Windows SCM；独立 migration、滚动升级、加密 Restic 备份、空环境整站恢复、恢复库切换和主密钥轮换均有真实演练 |
| 观测 | 公网不暴露 `/metrics`；backend 逐实例指标、稳定异常事件、Prometheus 阈值规则、Grafana 看板和对应 runbook |
| 成本 | 管理员维护不可变模型价格版本；请求冻结价格并按 authoritative/estimated usage 结算整数 nanos，按用户、套餐、模型、Provider、资源域和币种聚合；成员不可读取采购价 |

## 公司如何形成收入

LLMGateway 本身不是收银台。公司通过 B2B 服务合同、团队席位或约定的模型用量套餐向签约客户收费，销售和财务在线下完成合同、开票与收款；管理员再按合同在网关内分配成员、模型权限、额度和期限。

收入来自统一接入、稳定运营、权限治理和模型资源整合所提供的服务价值；成本来自上游模型、基础设施和运维。网关账本记录可审计的使用事实、额度消耗和冻结的上游采购成本；客户售价、合同、毛利口径、开票和收款由公司经营流程拥有，不写死在 Provider adapter 或路由代码中。

## 日常运营闭环

```text
签约或内部审批
  -> 管理员邀请并审核成员
  -> 分配模型、额度、速率和 Key
  -> 成员调用统一 API
  -> 管理员查看 usage、余额与凭据健康
  -> 按合同调整额度、停用成员或撤销 Key
  -> Provider 变化时捕获并发布新目录
```

## 产品蓝图

### 当前交付

闭合 200～300 名受控用户的完整管理员与成员主旅程，证明统一 API、权限额度、路由限流、取消恢复、安全、备份恢复、主密钥轮换和一个真实部署目标。

### 稳定运营

根据真实 usage、Provider 错误和人工支持记录改进现有双核心：补强必要 adapter、容量参数、恢复工具和管理员工作流。

### 需求驱动扩展

当持续客户需求能证明收益大于长期维护成本时，再评估新 Provider、外部管理 API、企业身份或更强部署拓扑。每次只增加一个有 owner、有恢复、有验收的生产切片，不改变双核心定位。

## 完成判断

代码存在、mock 通过或一次顺利演示不等于完成。交付必须连接真实 Go、PostgreSQL、Valkey、生产构建前端和隔离 Provider，以有头 Chromium 覆盖桌面与移动管理员/成员主旅程，并验证网络、DOM、控制台、持久状态、排队、限流、取消、断连、刷新、重登、强杀和重启；备份恢复、主密钥轮换和真实部署目标分别提供演练证据。
