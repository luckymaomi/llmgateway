# LLMGateway 完整功能版图

本文档是需求讨论底稿，只说明产品可以解决什么问题，不决定技术栈、实现方式或开发顺序。

## 当前需求起点

目前已经确认的真实场景是：

- 管理者拥有多个合法取得的免费 API Key；不同 Key 对应不同 Provider、模型、额度和限流规则。
- 单个 Key 可能遇到 RPM、Token、并发、每日额度、超时和临时服务错误。
- LLMGateway 对外提供一个统一入口，把请求合理分配到仍然可用的 Key 和 Provider。
- 初期至少要支持几十人共同使用，由管理者给不同用户分配可用模型和额度。
- 核心目标是提高免费额度利用率、降低使用成本，并减少手动切换 Key 的工作。
- 未来可能开放用户注册、额度分配和成本分摊，但是否加入自助购买、充值和支付尚未确认。
- 可能收取少量服务器与运维成本，不等于已经确定转售上游 API；实际运营方式必须遵守各 Provider 的服务条款。
- 未来可能加入代理和 IP 出口管理，用于合法的跨地区访问、故障备用或稳定出口；具体调度方式尚未确认。

## 名词解释

### Provider

提供模型 API 的厂商或服务，例如智谱、DeepSeek、OpenAI-compatible 平台。一个 Provider 可以有多个模型和多个账号。

### API Key

Provider 发给用户的一串访问凭据。最简单的接入方式就是把合法取得的 API Key 保存到网关，由网关代为调用上游。

### 上游账号

网关用来调用 Provider 的身份。它通常只是一个 API Key，也可能包含组织、项目、区域或其他厂商要求的信息。

### OAuth 或订阅账号

它不是普通 API Key，而是通过账号登录授权、访问令牌或订阅权益调用服务。这类账号需要处理登录授权、令牌刷新、到期和平台条款，复杂度与风险明显更高。目前没有确认 LLMGateway 必须支持它。

### LLMGateway Key

LLMGateway 发给调用者的 Key。调用者只接触这个 Key，不需要知道网关内部保存了多少上游 Key。

### RPM、TPM 与并发

- RPM：一分钟最多允许多少次请求。
- TPM：一分钟最多允许使用多少 Token。
- 并发：同一时刻最多允许多少个请求正在执行。

### 负载均衡

把请求分配给多个可用账号，避免所有流量集中到一个 Key。它可以依据优先级、权重、额度、速度、成功率和当前负载做选择。

### 会话粘性

同一段连续对话尽量继续使用同一个上游账号。某些模型会在上游保存会话状态，随意切换账号可能导致上下文断裂。

### 代理与 IP 出口

网关访问上游时经过的网络出口。专业的出口管理通常关注稳定绑定、地区、延迟、健康、到期和备用，而不是无目的地频繁更换 IP。

### 重试、冷却与熔断

- 重试：临时失败后再次尝试。
- 冷却：某个账号刚刚限流或失败，暂时不再使用它。
- 熔断：某个账号或 Provider 持续异常时整体暂停调用，避免不断制造失败。

## 完整功能版图

下面列出成熟 LLM 聚合网关可能具备的完整能力。它们是需求候选，不表示首期必须全部实现。

### 1. Provider 与模型接入

1. **通用 OpenAI-compatible 接入**：填写 Base URL、API Key 和模型信息，就能接入遵循通用协议的 Provider。
2. **厂商专用接入**：对确实存在私有协议的 Provider 做独立适配，而不是要求用户自己转换格式。
3. **多凭据管理**：同一 Provider 可以添加多个 API Key、账号、项目或区域。
4. **模型目录**：展示每个模型来自哪里、支持什么能力、当前是否可用。
5. **模型别名**：对外使用稳定名称，内部可以映射到不同 Provider 的真实模型名。
6. **能力匹配**：明确模型是否支持流式、工具调用、思考内容、图片、视频、语音、长上下文和结构化输出。
7. **参数转换**：把统一请求转换成各厂商能够理解的参数，并明确暴露无法转换的能力。
8. **Provider 测试**：添加账号后可以测试连通性、模型可用性、响应速度和错误原因。
9. **Provider 特定接口透传**：统一协议覆盖不了的批处理、文件或其他接口，可以在权限控制下透传。

### 2. 统一调用入口

1. **一个 Base URL**：调用者只配置一个地址。
2. **一个 LLMGateway Key**：调用者不接触上游真实密钥。
3. **标准模型列表**：客户端可以查询自己有权限使用的模型。
4. **Chat Completions**：兼容常见聊天客户端和 Agent。
5. **Responses**：支持需要状态、工具和新事件格式的客户端。
6. **流式输出**：模型生成一部分就向客户端返回一部分，并正确处理断开和超时。
7. **工具调用**：统一不同 Provider 的 Function Calling 表达。
8. **思考内容**：在 Provider 允许的范围内统一 reasoning 字段，不把它混入普通正文。
9. **多模态接口**：逐步覆盖图片、视频、语音、Embedding 和 Rerank。
10. **统一错误格式**：客户端不需要理解每家 Provider 完全不同的错误结构。

### 3. 上游账号池

1. **账号分组**：按 Provider、模型、用途、团队或服务等级组织账号。
2. **状态管理**：区分正常、禁用、限流、过载、到期、异常和冷却中。
3. **额度记录**：记录能够查询到的余额、每日额度、重置时间和实际用量。
4. **未知额度**：Provider 不提供余额查询时明确显示未知，不伪造剩余额度。
5. **到期管理**：账号或 Key 临近到期时提醒，到期后停止调度。
6. **批量导入与检查**：方便管理多个合法 Key，但不包含批量注册、接码或养号。
7. **密钥轮换**：替换泄漏或失效的 Key，不影响调用方的 LLMGateway Key。
8. **人工控制**：管理员可以暂停、恢复、测试和备注账号。
9. **OAuth 刷新**：只有确认需要接入 OAuth 账号时，才考虑自动刷新访问令牌。

### 4. 智能调度与负载均衡

1. **固定优先级**：优先使用指定 Provider 或免费额度，失败后再使用备用资源。
2. **轮询**：多个条件相同的账号依次使用。
3. **加权分配**：性能更好或额度更多的账号获得更多请求。
4. **按剩余额度选择**：优先使用仍有充足额度的账号，避免某个账号过早耗尽。
5. **按当前负载选择**：优先选择并发较低、排队较少的账号。
6. **按成功率选择**：近期成功率高的账号获得更多流量，同时给新账号保留试用机会。
7. **按响应速度选择**：在能力一致时优先选择延迟较低的账号。
8. **按成本选择**：免费或低成本资源优先，昂贵资源只作为备用。
9. **能力路由**：工具调用、图片、视频或长上下文请求只能进入支持对应能力的账号。
10. **会话粘性**：同一会话尽量使用同一账号，失效时按明确规则迁移。
11. **条件路由**：按用户、团队、模型、请求类型、地区或服务等级选择不同账号池。
12. **灰度流量**：新 Provider 或新模型先接收少量请求，确认稳定后再扩大流量。
13. **可解释结果**：管理员能看到为什么选择了这个账号，而不是只能猜测调度器行为。

### 5. 限流、排队与公平使用

1. **上游账号限流**：分别控制每个账号的 RPM、TPM、并发和日额度。
2. **Provider 总限流**：多个账号仍可能共享厂商总限制，必要时限制整个 Provider。
3. **调用方限流**：按用户、团队、LLMGateway Key 和模型限制请求速度。
4. **并发限制**：一个用户不能占满所有长连接和生成任务。
5. **额度限制**：给用户或团队设置日、月或总额度。
6. **短时排队**：资源暂时繁忙时允许有限等待，而不是立即失败。
7. **公平调度**：防止一个高频用户长期占用全部账号池。
8. **优先级队列**：正式业务、管理员任务和普通测试可以拥有不同优先级。
9. **明确拒绝原因**：告诉调用者是用户额度不足、账号池饱和还是 Provider 限流。
10. **防止无限等待**：排队必须有长度和时间上限。

### 6. 重试、故障切换与恢复

1. **错误分类**：区分限流、超时、网络故障、认证失败、余额不足、参数错误和内容拒绝。
2. **有限重试**：只对明确的临时错误重试，并限制次数和总等待时间。
3. **尊重上游等待时间**：Provider 返回 Retry-After 时按其要求等待。
4. **阶梯退避**：连续失败时逐步延长等待，避免故障期间继续轰炸上游。
5. **备用账号切换**：当前账号不可用时选择同能力账号。
6. **跨 Provider 回退**：只有模型能力和业务语义兼容时，才切换到其他 Provider。
7. **账号冷却**：429 或过载后暂时排除账号，到期后自动恢复。
8. **熔断与试探恢复**：持续故障时停止调用，恢复前先用小流量测试。
9. **流式边界**：内容已经发送给客户端后，不能悄悄从头重试并产生两份输出。
10. **未知结果保护**：无法确认上游是否已经执行的请求，不盲目重放。
11. **完整重试链记录**：管理员可以看到一次请求经历的全部尝试和最终结果。

### 7. 代理与 IP 出口管理

1. **HTTP/HTTPS/SOCKS 代理**：保存合法可用的网络出口。
2. **账号固定出口**：需要稳定环境的账号可以长期绑定同一个代理。
3. **地区信息**：记录出口国家、地区和城市，满足合法的区域访问需要。
4. **健康检查**：检查代理是否可连接、延迟多高、出口 IP 是否变化。
5. **质量评分**：依据成功率、延迟和稳定性判断出口质量。
6. **过期提醒**：代理临近到期时告警，到期后停止使用。
7. **备用代理**：主代理故障后可以切换到明确配置的备用代理。
8. **直连回退**：只有管理员明确允许时，代理失败才回退直连。
9. **地区路由**：某些 Provider 或模型可以选择指定地区的出口。
10. **有限轮换**：只有业务确实需要且符合 Provider 条款时，才按策略更换出口。
11. **安全限制**：代理和自定义地址不能被利用来探测服务器内部网络。

### 8. 用户、团队与权限

1. **管理员创建用户**：最简单的团队使用方式，由管理员添加成员。
2. **用户注册**：未来可以选择开放注册，但不是当前必做事实。
3. **团队与项目**：不同成员可以共享额度，也可以彼此隔离。
4. **LLMGateway Key 管理**：用户可以拥有一个或多个 Key，并设置名称和有效期。
5. **模型权限**：限制某个用户或 Key 能使用哪些模型。
6. **额度分配**：管理员给用户、团队或 Key 分配额度。
7. **速率与并发权限**：不同用户可以拥有不同服务等级。
8. **来源限制**：可选地限制允许调用 Key 的 IP 或网络范围。
9. **角色权限**：区分系统管理员、运营人员、团队管理员和普通调用者。
10. **登录安全**：根据规模逐步加入邮箱验证、2FA、OIDC 或企业 SSO。
11. **停用与回收**：成员离开后立即停用账号和 Key，同时保留必要审计记录。

### 9. 用量、额度与成本分摊

1. **请求账本**：记录谁在什么时间使用了什么模型和多少额度。
2. **Token 用量**：区分输入、输出、缓存和推理 Token。
3. **媒体用量**：图片按尺寸或张数、视频按时长、语音按时长或字符记录。
4. **真实值与估算值分离**：优先采用 Provider 返回的真实 usage，否则明确标记本地估算。
5. **价格版本**：价格变化后，历史账单仍按当时规则解释。
6. **免费额度利用**：显示每个账号已经使用多少，避免有的闲置、有的提前耗尽。
7. **用户额度**：按用户、团队、Key、模型和时间周期设置上限。
8. **预算提醒**：接近额度时提醒，而不是用完后才发现。
9. **成本分摊**：可以统计每个人应承担的服务器、运维或上游成本。
10. **人工分配**：初期由管理员直接增加或调整额度。
11. **自助充值**：只有确认需要面向外部用户运营时再考虑。
12. **支付与退款**：属于独立的高复杂度产品能力，当前没有确认必须实现。
13. **防重复扣费**：一次请求发生重试、超时或回调时不能重复记账。

### 10. 图片、视频与异步任务

1. **创建任务**：把长时间生成请求保存为可追踪任务。
2. **状态查询**：显示排队、处理中、成功、失败和取消。
3. **后台轮询**：Provider 没有主动回调时，由网关按合理频率查询。
4. **Webhook 回调**：Provider 或 LLMGateway 可以在任务完成时通知调用方。
5. **任务取消**：Provider 支持时可以取消，不能伪造已经取消。
6. **结果保存**：记录最终文件地址、有效期和必要元数据。
7. **重启恢复**：网关重启后继续跟踪已有任务，不重复提交。
8. **任务限流**：图片和视频使用独立并发与额度，避免拖垮聊天请求。
9. **任务计费**：按实际成功结果和 Provider 合同记录用量。

### 11. 管理后台与日常运营

1. **总览看板**：查看请求量、成功率、活跃用户、额度和当前故障。
2. **Provider 管理**：添加、编辑、测试和停用 Provider。
3. **账号池管理**：查看每个 Key 的状态、用量、限流和错误。
4. **模型管理**：维护别名、能力、权限和路由范围。
5. **用户与 Key 管理**：分配额度、权限、并发和有效期。
6. **代理管理**：查看出口 IP、地区、延迟、质量和到期时间。
7. **策略管理**：配置优先级、权重、回退、限流和冷却规则。
8. **请求查询**：按用户、Key、模型、Provider、错误和时间检索。
9. **批量操作**：批量测试、启停、分组和修改账号。
10. **配置审计**：记录谁在什么时候修改了什么设置。
11. **导入与备份**：安全迁移配置和数据，不导出明文密钥。

### 12. 监控、告警与排障

1. **请求追踪 ID**：一次请求的全部尝试可以串联查看。
2. **核心指标**：请求量、成功率、首 Token 延迟、总耗时、排队时间和错误率。
3. **账号健康**：展示限流、过载、余额不足、认证失败和恢复时间。
4. **代理健康**：展示出口变化、连接失败和延迟异常。
5. **额度告警**：账号、用户或团队接近上限时通知管理员。
6. **故障告警**：Provider 大面积失败、队列堆积或错误率升高时告警。
7. **脱敏日志**：保留排障事实，不记录真实 Key 和默认请求正文。
8. **状态页**：让使用者知道当前哪些 Provider 或模型不可用。
9. **自动处置**：按明确规则暂停异常账号、切换备用资源并在恢复后重新试探。

### 13. 安全与合规

1. **密钥加密**：上游 Key 和代理密码不能明文散落在配置和日志中。
2. **最小权限**：普通用户不能查看上游账号、其他用户或系统设置。
3. **日志脱敏**：Key、令牌、个人信息和敏感请求内容默认隐藏。
4. **管理审计**：密钥、额度、权限和路由策略的变更有记录。
5. **地址安全**：自定义 Base URL、代理测试和 Webhook 不能访问禁止的内部地址。
6. **请求大小限制**：防止超大正文、压缩包和媒体数据耗尽服务器资源。
7. **滥用控制**：限制暴力请求、Key 扫描和异常注册行为。
8. **密钥轮换和撤销**：发现泄漏时可以立即停用并替换。
9. **数据保留规则**：明确日志、账单、请求内容和媒体结果保留多久。
10. **Provider 条款检查**：接入方式、额度共享、代理使用和成本分摊都必须遵守上游规则。

### 14. 扩容、高可用与运维

1. **单机简单部署**：初期几十人可以低成本运行和维护。
2. **多实例扩容**：流量增加后可以增加网关实例，而不是重写产品。
3. **共享一致状态**：多个实例看到相同的账号状态、额度、限流和任务。
4. **无单点故障**：一个网关实例退出时，其他实例继续服务。
5. **优雅停机**：升级或重启时尽量完成已有流式请求，不突然截断。
6. **配置一致性**：策略变更能够安全发布到所有实例。
7. **数据库备份与恢复**：误操作、磁盘故障和升级失败后可以恢复。
8. **数据迁移**：版本升级时能够明确迁移数据，不依赖手工改表。
9. **容量保护**：队列、连接、日志和任务都有上限，不会无限占用资源。
10. **跨平台交付**：覆盖 Windows、Linux、macOS 和常见 x64/ARM 环境。

### 15. 后续扩展能力

1. **Provider 插件**：新增 Provider 时不需要修改整个系统。
2. **开放管理 API**：管理后台以外的系统也可以自动创建 Key、分配额度和读取用量。
3. **Webhook**：额度、故障和任务完成时通知其他系统。
4. **响应缓存**：对安全且可复用的请求减少重复调用。
5. **条件路由**：按业务、地区、成本和质量组合路由规则。
6. **灰度与模型评测**：比较新旧模型表现后再切换主流量。
7. **内容安全规则**：按实际需要增加输入输出检查。
8. **MCP 网关**：未来如果需要统一管理 Agent 工具，可以作为独立扩展，不与首期 LLM 转发强绑定。

## 需求优先级候选

### 当前核心候选

- 通用 OpenAI-compatible Provider 接入。
- 多 API Key 管理与加密保存。
- 统一 Base URL 和 LLMGateway Key。
- 模型别名、权限和能力匹配。
- 上游 RPM、TPM、并发与额度管理。
- 面向几十人的用户、Key、模型权限和额度分配。
- 基础负载均衡、失败切换、冷却和有限重试。
- 请求用量、账号健康和错误记录。
- 简洁可用的管理后台。

### 应当提前考虑但可以稍后完善

- 会话粘性和不可安全重试的边界。
- 多实例共享限流和账号状态。
- 图片、视频异步任务的恢复与防重复提交。
- 代理与账号的稳定绑定关系。
- 账本防重复、价格版本和历史可解释性。
- 管理操作审计、备份和密钥轮换。

### 尚未确认的可选能力

- OAuth 或订阅账号接入。
- 开放注册、邮箱验证和自助找回。
- 自助充值、支付、退款和订阅套餐。
- 对外商业运营与完整客户体系。
- 大规模 IP 轮换。
- 语义缓存、内容安全、MCP 网关和复杂模型评测。

## 后续逐项确认顺序

1. 允许接入哪些上游凭据：仅官方 API Key，还是包括 OAuth 或订阅账号。
2. 首批 Provider、模型和协议接口。
3. 用户如何加入：管理员创建、邀请注册还是完全开放注册。
4. 管理员如何给用户分配额度，以及是否需要成本分摊。
5. 负载均衡最重视什么：免费额度、稳定性、速度还是公平使用。
6. 哪些错误允许重试，哪些请求绝不能自动重放。
7. IP 出口的真实用途：稳定绑定、跨地区、故障备用还是有限轮换。
8. 是否需要图片和视频异步任务。
9. 日志保存什么、不保存什么，以及谁可以查看。
10. 初期几十人的实际并发、请求量和部署环境。
11. 哪些能力必须首期完成，哪些只保留未来扩展空间。
12. 需求稳定后，再判断从零实现、复用组件还是基于现有项目二次开发。

## 调研参考

- LiteLLM：统一 Provider、虚拟 Key、多租户预算、限流和多种路由策略。
  - https://github.com/BerriAI/litellm
- New API：渠道管理、用户、额度、计费、模型接口和集群部署。
  - https://github.com/QuantumNous/new-api
- Sub2API：上游账号池、粘性调度、并发控制、代理出口和用量账本。
  - https://github.com/Wei-Shaw/sub2api
- Portkey Gateway：可组合负载均衡、Fallback、重试、缓存和 Guardrail。
  - https://github.com/Portkey-AI/gateway
- Uni-API：轻量配置式 Provider 转换、轮询、冷却和限流。
  - https://github.com/yym68686/uni-api

这些仓库只作为需求和架构研究证据。正式实现前必须单独检查许可证和可复用边界；不能因为本地存在参考源码就直接复制到 LLMGateway。
