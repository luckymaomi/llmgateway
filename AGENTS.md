# BirdAPI Agent 工作规约

本文件是本仓库的最高工作约束。产品、架构与边界事实统一以 `spec.md` 为准；进入正式实现后，当前任务状态以 `plan.md` 为准。

## 基础约束

- 始终使用简体中文和项目 owner 交流，除非 owner 明确要求其他语言。
- 每次接手都先读取 `spec.md`、存在时的当前 `plan.md` 和命中的项目级 Skill，再做判断。
- 先调查当前仓库、运行结果和外部合同，再修改文件；用户建议和历史实现都只是证据。
- 不回滚 owner 已有改动，只处理当前任务范围内的文件。
- commit、push、发布和部署只在 owner 明确授权时执行。
- 禁止机械替换、无审查批量替换和大小写不受控的命名迁移。
- 不保留旧别名、兼容转发、历史包装或过渡文件；当前实现只服务当前生效事实。

## 产品主线

- BirdAPI 是面向多 Provider 的 LLM 聚合网关，优先接入合法可用的免费模型、免费套餐与免费额度。
- “免费”指优先接入用户可合法使用的免费模型或免费额度，不代表 BirdAPI 承诺上游永久免费、无限额度或永不变更。
- 目标主链路是：客户端标准请求 -> BirdAPI 鉴权与策略 -> Provider adapter -> 上游调用 -> 标准响应与运行记录。
- 技术栈、Provider 合同和数据模型必须在正式实现任务中基于事实确定，不能由初始化偏好代替设计判断。

## 事实纪律

- 事实只能来自 owner 明确输入、当前仓库、命令输出、真实运行结果或已核验的官方资料。
- 区分已确认事实、owner 提供但未验证的事实、推测、计划事项和已完成并验证事项。
- 任何未实现或未验证内容，不得在源码、测试、配置、README、spec 或计划中表述为已经实现。
- Provider 模型、额度、RPM、错误码和功能支持都属于易变外部事实，接入前必须按官方合同重新核验。
- 新事实推翻当前设计时，先更新 `plan.md` 与 `spec.md`，再继续实现。

## 文件职责

- `AGENTS.md`：长期工作规则与协作纪律。
- `spec.md`：当前产品事实、边界与验收标准。
- `plan.md`：正式实现阶段当前中大型任务的单文件执行合同；无实际实现任务时不创建。
- `README.md`：面向用户和贡献者的项目入口。
- `.agents/skills/`：项目开发与计划方法，不保存产品运行状态。

## Skills

- 修改实现、接口、数据、测试、事实文档、依赖或运行配置时，先读取 `.agents/skills/birdapi-dev/SKILL.md`。
- 正式实现阶段的中大型开发、跨模块修改、协议或安全边界调整、生产级验收时，同时读取 `.agents/skills/plan/SKILL.md` 并维护根目录 `plan.md`。
- 业务专属 Skill 只在真实重复工作流出现后新增，不把临时任务硬编码进通用 Skill。

## 架构纪律

- 协议入口、Provider adapter、凭据池、调度策略、状态存储、重试策略和可观测性必须各有明确 owner。
- 通用 OpenAI-compatible 合同由协议层维护；厂商差异只进入对应 adapter 或明确的 Provider policy。
- Key 的选择、限流、熔断和恢复由调度边界统一决定，调用方不得各自复制策略。
- 重试必须区分读操作、确定未发送的请求和副作用未知的请求；不得因切换 Key 而盲目重放未知请求。
- 配置、公共 schema、错误格式、事件和持久状态属于跨模块合同，修改时同步检查所有消费者、测试和文档。
- 单一职责按变化原因判断；文件超过 300 行时审查职责，但不按行数机械拆分。

## 安全边界

- 不读取、打印、提交或上传真实 API Key、令牌、个人数据和私有配置。
- 测试只使用明确的假密钥或隔离测试凭据；日志、错误与指标必须默认脱敏。
- 不实现批量注册、接码、养号、绕过风控、规避平台条款或突破上游额度的能力。
- 对外监听、权限扩大、生产数据变更和付费操作必须得到 owner 明确授权。

## 验证与交付

- 仓库进入正式实现并确定技术栈时，必须在同一提交中建立统一的完整验证命令并同步本文件与开发 Skill。
- 架构初始化至少运行 `git diff --check`、文件范围检查、占位项检查和 Project Inception 只读扫描。
- 后续实现先运行最接近改动的测试，再运行项目完整验证与核心路径检查。
- 交付必须说明完成事实、实际验证、未验证项和剩余风险，不用复跑掩盖不稳定测试。
