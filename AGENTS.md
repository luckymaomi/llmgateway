# LLMGateway Agent 工作规约

本文件是仓库最高工作约束。`spec.md` 拥有产品与系统事实，`dev.md` 拥有长期开发和验收细则；正式实现期间，根目录 `plan.md` 是当前任务唯一执行合同。

## 1. 产品骨架

LLMGateway 是服务约 200～300 名受控用户的中小型企业级多 Provider 网关。它不是玩具，也不扩张成没有真实需求、难以演进的大型平台。

只有两个业务核心：

1. **多 Provider 统一服务**：在官方合同和服务条款内，最大化每条合法凭据的 RPM、TPM、并发和周期额度利用率；以统一 API、硬资格过滤、管理员 priority、同级 weight、冷却、熔断、有界重试和安全恢复提供可靠服务。未知副作用、传输断连和已提交流绝不盲目重放，免费池耗尽不自动进入付费池。
2. **管理员与成员治理**：管理员治理成员、邀请、审核、模型权限、额度和 Gateway Key；成员只操作并读取自己的事实，直接访问管理员 URL 或 API 也不能越权。

三个横切原则贯穿双核心，但不另建业务事实：

- **用户友好**：Web、CLI 和文档只把唯一内核事实投影成自然任务，让首次使用者从当前状态直接看见下一步；不以帮助文案掩盖信息架构问题。功能存在但管理员或成员无法自然发现、理解、完成和排障，不算交付完成。
- **安全**：身份、secret、账本、网络、资源、供应链、部署、备份和恢复遵循最小权限、可审计、脱敏和失败关闭。
- **快速迭代**：Provider、模型、运营能力和界面会持续扩展；稳定业务合同不变时，页面、按钮、文案、布局、文件名和内部模块可以重建，不受测试或兼容壳反向束缚。

用户友好不是第三套业务内核或事实 owner，但与功能正确性同属交付门槛：

- 管理员和成员使用同一控制台框架，按 capability 展示不同任务；导航隐藏不能代替服务端权限，界面也不得另算身份、额度、健康、成本或调度事实。
- 信息架构按用户任务组织。管理员直接看见上游资源、成员服务、运营数据和系统设置；成员直接看见自己的 API Key、额度、请求记录和账号操作。Provider API Key 与 Gateway Key 必须始终使用不会混淆的名称和上下文。
- 首屏先回答当前是否可用、哪里需要处理和下一步可以做什么；列表行就地提供该对象最常用的查看、测试、更换或撤销动作，复杂详情再进入弹窗或独立页面。
- 全新、空、加载、失败、冷却、停用、待审核、待确认和已完成状态都必须有真实事实、明确反馈与可执行出口；不能用虚假健康、占位数据、万能设置页或大段帮助文本填补缺失流程。
- 桌面与移动端复用同一任务和合同，移动端导航必须可打开、可关闭且不遮挡内容；表格可在自身边界内滚动，页面主体不得横向漂移或靠缩小字体容纳。
- 参考项目只借鉴已验证的任务组织、状态反馈和操作闭环。支付、OAuth 账号池、粘性会话、iframe、分组或其他能力只有在当前产品拥有真实需求、唯一 owner 和完整安全边界后才出现入口，不先画菜单再补内核。

## 2. 接手与执行

1. 每次接手先完整读取 `AGENTS.md`、`spec.md`、`dev.md`、命中的项目 Skill 和存在时的 `plan.md`。
2. 检查 Git、未提交改动、入口、持久状态、测试、文档、运行证据和外部合同。用户输入、历史讨论、参考项目与绿灯都只是证据，不自动等于当前事实。
3. 先 research，再核验边界和唯一 owner，最后行动；未完成相关核心链路调查前不修改局部实现。
4. 正式实现先执行 `rapid-iteration`，再执行 `llmgateway-dev`；中大型或跨模块任务同时执行 `plan` 并实时维护 `plan.md`。
5. 保护 owner 的现有改动。commit、push、部署和发布只在 owner 明确授权时执行。

始终用简体中文与 owner 交流，仓库文本按 UTF-8 读取。PowerShell 调用 Node 包管理器使用 `npm.cmd`、`npx.cmd`、`pnpm.cmd`。

## 3. 事实与架构

沿真实链路调查并闭环：

```text
客户端 -> 公共协议 -> 鉴权 -> 权限/额度 -> admission/排队
       -> 路由/凭据 -> Provider adapter -> 上游发送
       -> 流式/非流式响应 -> usage 结算 -> 日志/指标/审计
```

管理面继续追踪写入、校验、发布、生效、回滚和数据面读取。修改任何事实前列出输入、消费者、持久状态、展示、错误、恢复、测试和文档。

- 每个事实只有一个 owner；handler、UI、adapter、store 和测试不得重算或保存第二套事实。
- Public API 拥有客户端合同，Canonical Model 拥有统一语义，Provider Adapter 或明确 policy 拥有厂商 wire 差异。公共协议、调度和账本不得散落厂商名判断。
- 通用 OpenAI-compatible 实现只承载语义和变化原因都相同的合同；无法无损表达的厂商能力留在独立 adapter/policy。
- Provider 模型、能力、错误、价格和额度是易变外部事实，修改前依据官方资料和隔离 wire 复核；无法权威查询的余额标为未知或估算，不伪造剩余额度。
- 派生展示可以组合权威输入，但不得成为新的调度、身份、额度、成本或健康 owner。

## 4. 实现与数据

- 每个纳入范围的切片都达到生产级闭环：成功、拒绝、持久化、并发、中断、恢复、安全、观测、测试、文档和目标环境验证同时成立。范围可以小，质量不能降级。
- 主干不接受 demo、占位成功、虚假余额/健康/支持、只在内存成立的关键事实、已知破损恢复、无验收 TODO 或准备以后删除的双轨实现。
- 证据推翻 owner、schema、公共合同或状态机时，修改唯一源并同步重建消费者；删除优先于兼容，不保留旧别名、转发、包装、双读双写、临时桥接或测试专用路径。
- 当前尚无必须保留的正式生产数据：首次生产发布前，不兼容变化直接重建基线。正式生产数据出现后才使用一次性、可验证、可回滚的前向迁移，完成后删除旧路径。开发/测试清理必须独立、显式且范围可核验。
- 单一职责按事实 owner 和变化原因判断；跨 owner 决策、事实重复、循环依赖、领域反向依赖交付层或文件超过 300 行时触发职责审查，不按行数机械拆分。
- 结构化数据使用结构化接口；错误类型服务恢复策略，不靠散落字符串控制核心流程。生成物必须可确定性重建且禁止手改。
- 配置集中、类型化并在启动时完整校验；无效配置失败退出，不保留旧 env 别名或静默回退。新依赖先核验必要性、许可证、维护状态、来源与目标平台。
- 命名按当前领域职责并暴露单位，不用 `v1/v2/v3`、`new/old`、`legacy/temp/final` 表达内部演进；外部协议版本、官方模型名和迁移序号除外。金额不用浮点数。

## 5. 失败、测试与验收

正常退出和稳定上游不是正确性的前提。每项能力检查接受、执行、提交、中断、失败、恢复、重放和清理，重点覆盖重复提交、并发竞争、取消、断连、超时、429/5xx、畸形流、凭据停用、存储/Valkey 短暂不可用、强杀、重启和多实例恢复。重复取消、恢复和清理必须幂等。

测试服从“稳定核心优先”：

- 测试存在前先通过三问：它保护两个业务核心中的哪个稳定结果；为什么现有更高层证据不能证明；模块、页面或文件重建后它是否仍有产品意义。任一项答不出就不新增，已有测试答不出就删除。测试名称和失败信息应描述业务结果或风险，不描述当前组件和内部步骤。
- 长期测试只保护可观察的业务结果、公共协议、Provider wire、权限、额度、账本、调度、幂等、并发、发送边界、安全和恢复。每个结果只保留最有证明力的一层；真实主旅程已证明的结果不再由 mock、组件、handler、service 和 repository 重复保护。
- 新增测试前先用一句业务语言说明它保护什么。若只能描述按钮、菜单、标题、文案、颜色、布局、文件、路由、调用次数、查询批次或内部步骤，则它不属于长期合同，应删除或提升到正确层级。
- UI 增删改名、重新排列或模块重建不触发测试，除非稳定业务结果、权限边界或公共合同变化。可访问名称可以定位真实操作，但定位方式不是验收结果；不以源码扫描、字符串不存在或目录形状证明完成。
- 浏览器按用户任务复用桌面与移动场景，连接真实 Go、PostgreSQL、Valkey 和生产前端；mock 只保留真实环境难以稳定制造的通用边界。测试等待可观察状态，不用固定 sleep 或复跑掩盖竞态。
- 视觉审美由人类验收；Agent 只验证结构、尺寸、不溢出、导航、表单、状态和交互。详细 UI 约束以 `dev.md` 为准。
- 测试超过一分钟不能只因为场景多或重复运行。它必须明确短测无法替代的时间相关或目标环境风险、最小有效持续时间、运行频率和失败归属；否则拆成定向短测、降低频率或删除。`everything` 只是发布候选时串联 full、真实 Provider、容量和发布物的组合入口，不是每次页面、Provider 或运营迭代的默认完成条件。
- Agent 默认只自行运行预计一分钟以内的定向测试。`daily`、`full`、真实 Provider、容量、发布物和 `everything` 由 owner 运行 `python .\start_test.py <mode>`；只有 owner 在当前任务明确授权指定长档时，Agent 才运行该统一入口并以 3～10 分钟间隔等待结果，不高频轮询。入口实时输出并写入 `.build/test-logs/`；失败从首个真实根因继续，不靠重跑掩盖不稳定。

## 6. 安全与交付

- 不读取、打印、提交或上传真实 API Key、令牌、个人数据和私有配置；日志、错误、指标、截图和测试夹具默认脱敏。
- 不实现批量注册、接码、养号、绕过风控、规避平台条款或突破上游额度。对外监听、权限扩大、生产数据变更和付费操作需要 owner 明确授权。
- 交付只陈述完成事实、实际运行证据、未验证项和外部风险；代码存在、mock 绿灯或一次测试通过都不能冒充产品完成。
