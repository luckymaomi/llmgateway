# LLMGateway Development Discipline

`dev.md` 约束 LLMGateway 如何开发，不描述尚未实现的产品能力。`spec.md` 保存当前产品事实，`architecture.md` 保存已确认技术与模块边界，Git 历史保存演进证据，正式实现任务由根目录 `plan.md` 管理。

## 1. 阶段交付标准

项目按生产级纵向切片推进。

每个阶段可以只交付一部分功能，但进入主干的范围必须端到端成立：真实入口可用，状态有唯一 owner，错误可判断，失败可恢复，安全边界明确，行为有测试，文档与实现一致，并在声明支持的环境完成验证。

以下内容不能作为阶段完成：

- 只跑通 happy path 的 demo；
- 用 mock 通过代替真实 wire contract；
- 把关键状态只留在进程内存；
- 以“后续补齐”为理由留下已知错误或双轨实现；
- 返回虚假成功、虚假余额、虚假健康或虚假支持；
- 只有代码，没有恢复、测试、文档和运行证据。

正式实现首先加载 `.agents/skills/rapid-iteration/SKILL.md`，并持续执行同一闭环：研究相关参考项目的许可证、机制与踩坑，核验 LLMGateway 当前事实与 owner，从真实入口复现失败，完成生产级纵向切片，再以有头 Chromium 的桌面/移动真实用户路径和相关恶劣路径验收。发现问题后回到参考机制与唯一 owner 修根因并复测，不能以第一次测试绿灯作为停止条件。

## 2. 当前事实优先级

1. 当前源码主链路与持久数据合同。
2. 当前 `spec.md`。
3. 当前确定性测试和生产路径验收。
4. 当前运行日志、数据库事实和外部 Provider 响应。
5. Git 历史与参考项目。
6. 用户偏好、旧讨论和第一判断。

历史用于解释取舍，不自动决定当前保留什么。参考项目用于研究机制，不拥有 LLMGateway 的产品语义。

## 3. 事实 owner

每类事实只允许一个 owner：

- Public Protocol：外部路由、请求校验、流式事件和标准错误。
- Canonical Model：消息、工具调用、推理内容、usage 和错误的内部语义。
- Identity and Access：用户、角色、邀请码、网关 Key、授权和停用。
- Provider Registry：Provider、模型、能力和端点配置。
- Credential Pool：上游凭据、授权范围、健康、冷却和禁用。
- Admission and Fairness：排队、用户并发、公平调度和背压。
- Routing and Scheduling：硬资格过滤、管理员优先级与同级权重选择。
- Rate Limit and Resilience：RPM/TPM、并发、超时、退避、重试、熔断和恢复。
- Usage and Quota Ledger：预留、权威 usage、估算、结算、退款和额度期限。
- Configuration Publication：管理面配置的写入、校验、发布、生效版本和回滚。
- Observability and Audit：非正文请求事实、稳定错误类别、指标、脱敏日志和管理审计。

Presenter、API handler、Provider adapter 和测试只能消费这些事实，不能复制第二套规则。

## 4. 核心链路

每次改动前按受影响范围追踪：

```text
请求接收
  -> 身份认证与权限
  -> 额度预留与 admission
  -> 模型路由与账号选择
  -> 限流与并发许可
  -> Provider wire 转换与发送
  -> 响应/流事件归一
  -> usage 结算或补偿
  -> 日志、指标、审计与用户结果
```

管理面继续追踪：编辑 -> 校验 -> 持久化 -> 发布版本 -> 数据面生效 -> 审计 -> 回滚。

任何 owner 合同变化，都要同步检查输入、消费者、持久状态、输出、错误、恢复、测试、配置和 `spec.md`。

## 5. Provider 与通用协议

- OpenAI-compatible 是公共复用合同，不是抹平所有厂商差异的借口。
- 厂商鉴权、端点、请求字段、流事件、工具调用、错误和 usage 差异进入 adapter 或明确 policy。
- 调度器只消费标准化能力与错误分类，不匹配厂商错误文案。
- 模型能力必须显式声明；不能无损转换时明确拒绝或暴露差异，不能静默丢字段。
- Provider 接入以官方文档和真实隔离请求为证据。模型名、价格、免费额度、RPM/TPM 和错误合同不得靠记忆固化。
- 复用开源 adapter 或协议代码前检查许可证、版本、维护状态和实际 wire 行为；机械复制不算复用。

## 6. 调度、额度与资金边界

- 免费池和付费池是不同资源域。免费池耗尽、冷却或不可用时返回明确状态，绝不隐式转入付费池。
- 免费池公平性由 admission owner 统一实现，至少能表达用户额度、并发上限、公平排队和共享闲置容量。
- 账号选择先做能力、资源域、启停、授权、冷却和本次 attempt 的硬过滤，再按最小管理员优先级和同级正权重选择；不伪造上游余额或综合评分。
- usage 账本区分预留、上游权威值、本地估算、最终结算和补偿。并发扣减必须原子化并可审计。
- Token Plan、Coding Plan 等产品规则由计划与额度 owner 表达，不能散落在 Provider adapter。
- 管理员分配额度必须形成可审计账本，不能直接修改一个无来源余额。

## 7. 错误与恢复

错误分类必须直接决定用户响应、重试、切换、冷却和熔断。至少区分：

- 客户端协议或参数错误；
- 身份、权限或额度错误；
- admission/限流/排队超时；
- Provider 配置或永久合同错误；
- Provider 临时限流、超时和 5xx；
- 上游发送边界未知；
- 流式响应部分完成；
- 持久存储、协调设施或配置发布失败；
- 主动取消、客户端断连和进程中断；
- 内部不变量破坏。

重试前必须知道请求是否可安全重放。未知副作用进入明确的 uncertain 状态，不能仅因换 Key 或换 Provider 就重放。错误 message 用于展示，稳定 error kind 才能参与控制流。

## 8. 中断、崩溃与恶劣用户路径

正常退出不是正确性的前提。假设用户会重复点击、重复提交、连续取消、关闭客户端、杀进程、重启主机，并在管理操作执行一半时再次操作；同时假设上游、网络和基础设施会在任何边界失败。

每项功能必须回答：

- 请求何时算被接受，接受事实何时持久化；
- 额度何时预留、结算、释放，进程中断是否会多扣或漏扣；
- 上游是否已经收到请求，未知边界如何记录；
- 客户端断连是否取消上游、继续结算，还是转为后台任务；
- 流已输出一部分后失败，用户、usage 和重试分别如何处理；
- 凭据或用户在请求中途被停用时，当前请求和后续请求如何分界；
- 多个并发请求是否可能突破用户、Key、账号或全局限额；
- 进程强杀、主机重启、持久存储或协调设施短暂不可用后，由哪个 owner 恢复；
- 重复取消、恢复、回调、轮询和清理是否幂等；
- 用户连续执行互斥配置变更时，数据面最终只会看到哪个合法版本；
- 失败是否留下可行动证据，而不是永久 running、静默丢失或虚假成功。

实现与测试至少覆盖：重复请求、客户端断连、429/5xx、超时、畸形响应、部分流、并发额度竞争、账号冷却、进程强杀恢复、配置发布竞态，以及免费池与付费池隔离。

## 9. 数据、兼容与迁移

- 先判断错误位于实现还是事实模型。局部实现违背正确合同时，在根因处修复；owner、schema、API 或状态机本身错误时，修改唯一源并完整重建/再生成受影响消费者，删除旧路径，不叠加补丁、例外判断或兼容层。
- 当前阶段尚无正式生产 schema 和必须保留的生产数据。数据库、公共 API、配置、事件或生成合同发生不兼容变化时，直接重建当前初始化事实，并在同一次交付中同步服务端、管理端、脚本、测试、样例和文档。
- 首次生产发布前不为中途设计积累迁移历史。已经被新事实推翻的 migration、字段、端点、配置键和适配层直接删除或压平为当前基线，不让开发数据成为兼容理由。
- 开发/测试数据重建使用独立、显式、范围可核验的命令，只能删除 LLMGateway 自己的容器、卷或测试数据库。普通启动、验证、依赖升级和应用迁移不得隐式清空数据。
- 当前 schema 只服务当前产品事实，不为尚不存在的旧版本建立兼容层。
- 正式生产数据出现前，不兼容变更直接重建测试和开发数据。
- 只有 owner 明确确认已有必须保留的正式生产数据后，才切换到生产迁移纪律。此后 schema 变化必须提供一次性、前向、可验证并可回滚的迁移；迁移完成后删除旧读写路径，不长期双写或双读。
- Git 历史和发布记录保存演进证据；运行时代码、当前 schema、测试和文档只表达当前生效事实。
- 持久记录需要稳定 ID、状态机、时间语义和审计来源。状态转换必须由事务或等价原子边界保护。
- 协调缓存可以丢失，权威事实不能只存在于缓存。恢复必须从持久 owner 重建。
- 普通日志、审计和指标不保存请求或响应正文，保留期由部署配置和运维边界显式管理。

## 10. 工程卫生与供应链

- 内部符号和目录按领域职责命名，不按实现先后命名。禁止用 `v1/v2/v3`、`new/old`、`legacy`、`temp`、`final` 或相似后缀表达“现在用哪套”。
- 外部 `/v1` 协议、Provider 官方模型名、标准版本号和一次性迁移序号必须忠实保留；它们不能扩散成内部版本化模块或双轨实现。
- 重命名先确认事实 owner、公共消费者、配置、持久数据、生成物、测试和文档，再逐文件语义修改。禁止无审查批量替换、大小写不受控迁移和只改声明不改消费者。
- 稳定领域合同使用显式类型。`map[string]any`、任意 JSON 和字符串键值只允许停留在确需透传的 wire 边界，进入核心判断前必须验证并归一化。
- 时间点统一使用 UTC 持久化，展示时再转换时区；持续时间、截止时间和重试时间使用不同类型或明确单位。测试通过可注入时钟、随机源和 ID 生成器控制非确定性。
- Token、请求数、并发、字节、毫秒和货币最小单位在类型或名称中显式表达。金额和费率使用整数最小单位或精确定点表示，不使用浮点数参与账本结算。
- 配置只有一个类型化 owner，启动时一次性校验必填项、组合约束、范围和敏感值来源。无效配置失败退出，不静默降级、不保留旧 env 别名、不在模块内散落重复默认值。
- 生成物不可手改。SQL、协议、前端路由或其他生成代码必须记录生成源与确定性命令，并在完整验证中检查生成漂移。
- 依赖只在减少真实复杂度或提供成熟机制时引入。引入前核验许可证、维护状态、发布来源、安全记录、跨平台支持和传递依赖；版本进入锁文件，升级与删除同步验证。
- 生产管理端资源由仓库构建并随发布物交付，不从 CDN 或运行时第三方脚本加载会接触凭据、请求内容或管理权限的代码。
- 注释只记录约束、原因和非显然取舍。TODO 必须有当前 `plan.md` 任务或可追踪 owner 与验收条件；没有落点的 TODO、占位成功、死配置和注释掉的旧实现直接删除。
- 模块依赖遵循 `architecture.md` 的 owner 方向；领域规则不反向依赖 HTTP handler、UI、Provider wire 或具体存储实现，不建立循环依赖。
- 模块或文件一旦出现第二个独立变化原因、跨 owner 判断、同一事实重复计算、领域层反向依赖交付/基础设施，或超过 300 行，就必须做职责审查。审查按 owner、输入输出和变化原因决定保持、收窄、移动或拆分，不按行数机械切割。

## 11. 安全规则

- 上游密钥、网关 Key、请求内容、用户数据和账本数据按敏感数据处理。
- 密钥只在最小必要边界解密，日志、错误、指标、测试夹具和追踪默认脱敏。
- 自定义上游 URL 必须防御 SSRF、恶意重定向、DNS 重绑定和内网地址访问。
- 公共接口设置请求体、上下文、并发、流时长和上传下载边界，避免无界资源消耗。
- 请求与响应正文不进入普通日志、审计、指标或持久存储。
- 不实现批量注册、接码、养号、绕过风控、规避平台条款或突破上游额度。

## 12. 测试与验证

测试预算先投向唯一核心：一个 owner 行为只选择最能稳定证明它的一层。真实主旅程证明跨边界可用，owner 定向测试证明浏览器无法充分制造的账本、权限、安全、幂等、并发、发送边界与恢复不变量；其余 handler/service/repository/component/mock 重复应删除。核心链未完成时，不建立页面级穷举、每层同义测试、桌面/移动两套脚本或 mock/真实两份完整业务旅程。

增加测试前必须说明它保护的稳定业务结果、现有证据为何不能覆盖，以及未来重构后仍有何产品意义。不能回答时不新增；发现多层测试保护同一结果时，保留证据最强且定位成本合理的一层，删除其余。浏览器场景按 capability 和用户任务参数化复用，选择器依赖可访问角色与稳定命令，不依赖组件嵌套、临时文案和内部数据形状。

长期测试优先保护：

- 公共协议与 Provider wire contract；
- 权限、额度和免费/付费隔离；
- 调度公平性、限流、冷却、熔断和有界重试；
- usage 原子结算与并发竞争；
- 流式请求生命周期；
- 持久状态、崩溃恢复、幂等和 uncertain 边界；
- 配置发布与数据面读取；
- 密钥脱敏、SSRF 和资源上限。

不长期保护：文件、路由、字符串或实现不存在，内部目录形状，关键词与源码文本，临时实现细节，纯文案和审美结果。测试正向验证核心业务结果和稳定合同，重构后仍应有产品意义；不用负性断言、文本扫描或测试数量冒充能力。

测试不依赖固定 sleep 或机器速度。并发测试使用可控同步点、事件或状态条件。真实 Provider 验收与日常确定性测试分层，真实验收不应成为每次本地验证的随机依赖。

技术栈确定后，仓库必须提供一个完整验证入口，覆盖格式、静态检查、单元、集成、竞态/并发、构建和必要的安全检查。目标平台差异必须在 CI 或明确的发布验证中覆盖。

所有需要审美判断的视觉结果由人类验收，Agent 不凭截图宣布颜色、间距、品牌感或整体观感合格。真实浏览器 Playwright 验收必须覆盖登录、权限导航、关键表单、配置发布、Key 一次展示、Playground 流式/取消/错误恢复，以及桌面和移动视口的结构、不溢出和可操作性。

管理端永久禁止单侧竖线式的伪 sidebar/callout：不得只在容器一侧放装饰边线，并让相对侧没有对应结构。强调与分组改用完整边界、背景、间距或排版；真实导航、工具和运行事实侧栏必须由实际内容与布局职责成立，不能靠一根线制造层级。

## 13. 收口

大改收口只写事实：

```text
现象：
根因：
修复：
验证：
未验证：
剩余风险：
```

不靠重复运行掩盖抖动，不把临时判断写成长期规则，不把历史失败写进当前产品主干。
