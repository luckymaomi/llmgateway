# LLMGateway Spec

`spec.md` 是 LLMGateway 产品、架构、边界与设计方向的唯一事实主干。

## 产品定位

LLMGateway 是面向多 Provider 的 LLM 聚合网关。它对外提供统一 API，并集中管理上游账号、模型、密钥、额度、限流、调度、故障切换和用量记录，优先接入合法可用的免费模型、免费套餐与免费额度。

“免费”指 LLMGateway 优先收集和接入用户可合法使用的免费模型、免费套餐或免费额度，不代表任何上游永久免费、无限额度或稳定不变。

## 核心体验

1. 管理员接入多个 Provider 及其独立上游凭据，配置模型、分组、优先级和限制。
2. 用户通过邀请码注册，由管理员审核、停用、授权模型并分配额度或套餐。
3. LLMGateway 为调用方签发统一的网关 API Key。
4. 调用方使用一个 Base URL 和一个 LLMGateway Key 访问已授权模型。
5. 网关完成鉴权、协议归一、模型路由、账号选择、公平排队、限流、重试和故障切换。
6. 管理端统一查看 Provider、账号池、模型、Key、用户、额度、用量、限流、错误和健康状态。

## 总体架构

```text
Client / SDK / Agent
        |
        v
Public LLM API -> 鉴权与请求归一 -> 路由与账号调度 -> Provider Adapter -> Upstream LLM
        ^                                                               |
        |                                                               v
        +------------ 标准响应 / 流事件 / 错误 / Usage -----------------+
```

- Public API 拥有客户端可见协议、鉴权、校验、流式生命周期和标准错误格式。
- Canonical Model 统一表达请求、响应、流事件、工具调用、推理内容、usage 和错误；无法无损转换的能力必须显式暴露。
- Provider Adapter 只处理对应厂商的鉴权、端点、协议转换、流解析、能力声明和错误归类；OpenAI-compatible Provider 复用通用 Adapter。
- Credential Pool 拥有上游账号与凭据的加密存储、模型授权、禁用、冷却和健康状态。
- Scheduler 与 Resilience Policy 统一拥有账号选择、并发、限流、超时、退避、重试、熔断、切换和恢复规则。
- Usage and Quota 区分上游权威 usage、本地计量和估算值；无法查询时明确标记未知，不伪造余额。
- Control Plane 管理 Provider、账号、模型、分组、LLMGateway Key、策略、用量、日志和健康状态，并向请求数据面发布已生效配置。
- Observability 记录脱敏请求 ID、路由结果、延迟、usage、限流、重试、熔断和错误类别。因滥用调查保留的请求内容属于独立受控审计数据，不进入普通日志。

## 产品能力方向

- OpenAI-compatible 的模型列表、Chat Completions、Responses 与流式接口。
- 统一 Function Calling、reasoning content、usage 和错误响应。
- Provider adapter 体系，容纳标准兼容接口和必要的厂商私有协议。
- 多账号与多 API Key 管理，支持模型分组、路由范围和独立策略。
- 全局、用户、Key、Provider 账号和模型维度的并发与速率限制。
- 基于健康、额度、RPM、并发、优先级与会话粘性需求的调度和故障切换。
- 共享免费池的用户额度、用户并发与公平排队；未使用容量仍由共享池调度。
- Token Plan、Coding Plan 和专业资源池的额度账本与期限模型；首期采用线下结算和管理员分配，不接支付接口。
- 有边界的重试、退避、熔断、恢复和长流式连接管理。
- 用量、请求、错误、限流与账号健康的脱敏记录和管理界面。
- Windows、Linux、macOS 以及常见 x64/ARM 环境的部署交付。
- 图片、视频和其他多模态接口在对应 Provider 合同明确后纳入统一网关表面。

## Provider 方向

首批接入范围是智谱 GLM、DeepSeek 和 Agnes，并保留通用 OpenAI-compatible Provider 接入能力。具体模型、鉴权、额度查询、RPM、流式、工具调用、图片/视频和错误合同必须在实现接入时依据官方资料验证。

当前范围不是已支持清单。无法可靠查询余额的账号不能伪造“剩余额度”，只能基于已知限额、实际 usage、响应状态和冷却时间维护可解释状态。

上游凭据由管理员统一管理。首期不接入用户自带 Key，不管理 OAuth 或第三方订阅账号，也不提供批量注册或账号获取能力。

## 资源域与用户

- 免费资源池优先聚合合法可用的免费模型、套餐和额度。
- 免费资源池与付费资源池严格隔离。免费资源耗尽、冷却或暂时不可用时返回明确状态并等待恢复，不自动消耗付费资源。
- 付费资源由管理员统一购买和配置，用户消费的是 LLMGateway 内部额度或套餐，不直接取得上游凭据。
- 用户注册必须持有邀请码，并经过管理员审核；管理员可以停用用户、撤销 Key、授权模型和调整额度。
- 第一阶段面向几十名用户，但公共协议、状态 owner 和部署边界不能阻塞后续水平扩容。
- 会话粘性只在模型能力、健康和资源策略允许时维持；它用于稳定路由与缓存收益，不假设上游保存客户端上下文。

## 边界

- LLMGateway 管理网关调用方与上游账号，但不提供批量注册、接码、养号、绕过风控、规避平台条款或突破免费额度的能力。
- 使用者负责合法取得上游凭据并遵守 Provider 的服务条款、额度和地区要求。
- 上游请求是否可重试由操作语义和发送事实决定；不能仅因切换账号就盲目重放未知副作用。
- 代理出口首期默认直连，并允许按 Provider 或上游凭据配置固定代理、健康检查和备用出口；不建设复杂轮换 IP 池。
- 支付、发票、公开售卖和自动充值不是首期既定能力；线下结算只通过可审计的管理员操作进入内部账本。

## 数据与安全

- 上游凭据、LLMGateway Key、账户状态和请求内容按敏感数据处理。
- 密钥不得进入 Git、普通日志、错误详情、公开 Issue 或默认遥测。
- 日志和指标默认脱敏，并区分请求事实、路由决策、usage、限流和错误分类。
- 请求内容可以为滥用调查进入受控审计存储，但必须有独立权限、访问审计、可配置保留期和删除规则。
- 凭据加密、权限模型、轮换、审计和数据保留周期必须在数据模型确定时成为显式合同。

## 工程与交付不变量

- 当前仓库尚无正式生产 schema 和必须保留的生产数据。首次生产发布前，不兼容的数据、API、配置和事件变化直接重建当前基线，不建立兼容层或过渡 migration；正式生产数据边界由 owner 明确确认后切换。
- 服务端主体使用 Go 1.26.5 或更新的 1.26 补丁版；采用模块化单体，首期不预拆微服务。
- PostgreSQL 18 是持久事实 owner，Valkey 9 只承担可重建的协调、限流、租约与缓存。
- 管理端使用 React 19、TypeScript 与 Vite 8，生产静态资源嵌入 Go 二进制；详细技术和模块边界以 `architecture.md` 为准。
- 支持 Windows、Linux、macOS 与常见 x64/ARM 目标；每个发布阶段只声明已经构建和验证的目标。
- 项目按生产级纵向切片推进。阶段可以缩小功能范围，但纳入范围的成功、错误、并发、中断、恢复、安全、可观测性、测试和文档必须闭环。
- 一个事实只有一个 owner；协议壳、管理界面和 Provider adapter 不拥有第二套状态。
- 主干不保留旧别名、兼容转发、历史包装、双 schema 读写或临时过渡路径。
- 未实现能力明确缺席，不以占位成功或模拟数据冒充支持。

## 正式实现前必须确认

- 最低生产运行环境、TLS/反向代理边界与各目标平台的发布验收方式。
- 第一阶段公共协议表面、三个 Provider 的已核验合同和管理端最小闭环。
- Provider/账号/模型/Key/策略/usage 的数据模型。
- 调度公平性、粘性会话、限流、重试、熔断和恢复语义。
- 网关鉴权、管理员权限、密钥加密、日志脱敏和数据保留规则。
- 第一个生产级切片的恶劣路径矩阵、完整验证入口和目标平台验收方式。
